# SKE + Cortex Integration Test Design

## Goal

Stand up Syntasso Kratix Enterprise (SKE) on minikube and validate the Cortex integration end-to-end: a labeled Promise automatically generates Entity Types and Workflows in Cortex, and triggering a Workflow provisions a resource on the cluster.

## Architecture

All components run on a single **minikube** cluster:

1. **SKE Operator + Kratix** — installed via Helm chart from `syntasso/ske-operator`
2. **SKE Cortex Controller** — enabled in the Helm values, configured with Cortex API key and the GitOps repo
3. **ArgoCD** — syncs Resource Requests from `jeff-test-org/ske-test-resources` (GitHub) to the cluster
4. **ConfigMap Promise** — a trivial Promise labeled `kratix.io/cortex: "true"`

### Data Flow

```
Promise installed on cluster
  → SKE Cortex Controller detects kratix.io/cortex=true label
  → Controller creates Entity Type + Create/Update/Delete Workflows in Cortex
  → User triggers Create Workflow via Cortex API
  → Cortex Workflow writes Resource Request YAML to jeff-test-org/ske-test-resources
  → ArgoCD syncs repo to cluster
  → Kratix processes Resource Request, runs pipeline
  → Pipeline outputs a ConfigMap
  → ConfigMap appears on the cluster
```

## ConfigMap Promise

### API

- **Group**: `demo.cortex.io`
- **Kind**: `Message`
- **Field**: `spec.message` (string)

### Pipeline Output

The resource configure pipeline reads `spec.message` and generates:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: <resource-request-name>
  namespace: default
data:
  message: <spec.message value>
```

### Why ConfigMap

- Zero dependencies: no operators or CRDs needed on the destination cluster
- ConfigMaps are built-in to every Kubernetes cluster
- Trivial to verify: `kubectl get configmap <name> -o jsonpath='{.data.message}'`
- Validates the full integration flow with minimum moving parts

### Scaffolding

Use the Kratix CLI to generate the Promise structure:

```bash
kratix init promise message --group demo.cortex.io --kind Message
kratix update api --property message:string
kratix add container resource/configure/message-configure \
  --image ske-test/message-pipeline:v0.1.0 --language python
```

The pipeline container image is built locally and loaded into minikube via `minikube image load`.

### Cortex Label

```yaml
metadata:
  labels:
    kratix.io/cortex: "true"
```

## Environment Variables

New variables added to `.env` / `.env.example`:

| Variable | Description |
|---|---|
| `SKE_LICENSE_TOKEN` | Syntasso license token (ghp_ prefixed, also used as GHCR pull secret) |
| `SKE_VERSION` | SKE version to install (e.g., `v0.99.0`) |

Reused from existing `.env`:

| Variable | Description |
|---|---|
| `CORTEX_API_KEY` | Cortex API key for the controller |
| `CORTEX_BASE_URL` | Cortex API base URL |
| `GH_USER` | GitHub username (for GHCR image pull secret) |
| `GITHUB_TOKEN` | GitHub token (for GHCR image pull + repo operations) |

## Justfile Recipes

### `ske-setup`

1. Start minikube if not running (reuse existing instance, same as k8s-agent)
2. Create GHCR image pull secret using `SKE_LICENSE_TOKEN`
3. Install ArgoCD into the cluster
4. Create GitHub repo `jeff-test-org/ske-test-resources` if it doesn't exist
5. Configure ArgoCD Application to watch the repo
6. Add Syntasso Helm repo and install SKE operator with Cortex controller enabled
7. Wait for SKE operator and Kratix pods to be ready
8. Build the pipeline container image and load into minikube
9. Install the ConfigMap Promise (with `kratix.io/cortex: "true"` label)
10. Wait for Promise status to become "Available"

### `ske-test`

1. Verify the Cortex Entity Type "Message" was created by the SKE controller
2. Verify Create/Update/Delete Workflows exist in Cortex
3. Trigger the Create Workflow via Cortex API with `{"message": "hello from ske"}`
4. Poll the cluster for the resulting ConfigMap (up to 5 minutes, accounting for GitOps sync time)
5. Verify ConfigMap contains the expected message
6. Report success/failure

### `ske-stop`

1. Uninstall the ConfigMap Promise
2. Uninstall SKE operator via Helm
3. Uninstall ArgoCD
4. Stop minikube

## Helm Values (SKE + Cortex Controller)

```yaml
skeLicense: <from SKE_LICENSE_TOKEN env var>
skeDeployment:
  version: <from SKE_VERSION env var>
cortexIntegration:
  enabled: true
  version: "latest"
  config:
    branch: main
    integrationAlias: github  # must match the GitHub integration alias in Cortex
    path: kratix-resource-requests
    provider: github
    repositoryName: jeff-test-org/ske-test-resources
    token: <from CORTEX_API_KEY env var>
    url: <from CORTEX_BASE_URL env var>
```

## Success Criteria

The test passes when all of these are true:

1. SKE Operator and Kratix pods are running
2. SKE Cortex Controller created a "Message" Entity Type in Cortex
3. SKE Cortex Controller created Create/Update/Delete Workflows in Cortex
4. Triggering the Create Workflow via Cortex API results in a ConfigMap appearing on the cluster
5. The ConfigMap's `data.message` matches the value sent in the Workflow

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| ArgoCD sync delay makes test flaky | Use generous polling timeout (5 min) in ske-test |
| SKE Cortex Controller version incompatibility | Pin `cortexIntegration.version` in values.yaml |
| GitHub repo permissions | Use `GITHUB_TOKEN` with repo create scope; `gh repo create` handles auth |
| Pipeline image not found in minikube | Use `minikube image load` instead of registry push |
